Implement DKIM and domain verification token TXT parsers, completing `src/utils/txt-parsers.ts`.

## Context

From the design document, section **Output Format > TXT Record Parsing**:

**DKIM** — detected by the `v=DKIM1` prefix. Parses key type, public key, hash algorithms, service types, and flags.

**Domain verification tokens** — detected by common prefixes. Known prefixes: `google-site-verification=`, `facebook-domain-verification=`, `MS=` (Microsoft 365), `atlassian-domain-verification=`, `apple-domain-verification=`, `_github-pages-challenge-`, `stripe-verification=`, `postmark-verification=`. Maintained as a simple prefix->provider map.

**Unrecognized TXT records** return `"parsed": null`.

## Tasks

- Implement DKIM parser: detect `v=DKIM1` prefix, parse key type, public key, hash algorithms, service types, flags
- Handle multi-string TXT records (DKIM keys often span multiple 255-byte strings)
- Implement verification token parser: detect known provider prefixes and extract the token
- Implement the prefix->provider map for all 8 known providers listed in the design document
- Ensure unrecognized TXT records return `parsed: null` without error
- Add to the detection chain after SPF and DMARC parsers
- Write unit tests (TDD) per the **Testing Strategy > Unit Tests > txt-parsers.ts** section:
  - DKIM: parse key type, public key, hash algorithms, service types, flags
  - DKIM: handle multi-string TXT records
  - Verification tokens: match each known provider prefix and extract the token
  - Unrecognized TXT: return `parsed: null` without error
  - Malformed DKIM (correct prefix, invalid body): return `parsed: null` with `parseError`

## Acceptance Criteria

- DKIM output matches the example structure in the design document
- Verification token output matches the example structure in the design document
- All 8 known provider prefixes are recognized
- Unrecognized TXT records produce `parsed: null`
- Detection chain short-circuits on first match
- All unit tests pass
