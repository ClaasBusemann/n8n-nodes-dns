Implement the UDP DNS client in `src/transport/dns-client.ts`.

## Context

From the design document, section **Transport Layer > UDP Client**:

> The client manages UDP socket lifecycle, timeouts, retries, and parallel dispatch to multiple servers.

> Each query creates a `dgram` UDP4 socket, sends the encoded packet, and listens for a response matching the transaction ID. On timeout, the socket is closed and the query retried up to the configured limit. When querying multiple servers, all queries are dispatched concurrently using `Promise.allSettled()` so that a slow or failing server does not block others.

> If the response has the TC (truncated) flag set, the node includes a `truncated: true` flag in the output but does not fall back to TCP in v1.

From section **File Responsibilities**:

> **`dns-client.ts`** — The single point of contact with the network. Creates `dgram` UDP sockets, sends encoded packets, listens for responses, handles timeouts and retries. All node-level code calls this module instead of touching `dgram` directly.

## Tasks

- Implement `dnsQuery()` that sends a DNS query to a specified server IP and port via UDP
- Create `dgram` UDP4 socket per query, handle socket lifecycle (create, send, receive, close)
- Match responses by transaction ID; discard mismatched responses
- Implement configurable timeout (default 5000ms) and retry count (default 1)
- Implement parallel multi-server dispatch using `Promise.allSettled()`
- Detect and flag TC (truncated) responses with `truncated: true`
- Write unit tests mocking `dgram.createSocket` with fixture buffers

## Acceptance Criteria

- All node-level code uses this module instead of `dgram` directly
- Timeout and retry behavior works correctly
- Multi-server queries run concurrently, not sequentially
- Transaction ID validation prevents response spoofing
- Truncated responses are flagged, not errored
