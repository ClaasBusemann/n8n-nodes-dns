Implement item linking (`pairedItem`) and error handling across both nodes.

## Context

From the design document, section **Item Linking**:

> Because this is a programmatic-style node, n8n does not automatically track item provenance. The node must set `pairedItem` on every output item.

Rules:
- **Single-server lookup (1:1):** Output item at index `i` sets `pairedItem: { item: i }`
- **Multi-server lookup (1:1 combined):** Combined results for input item `i` sets `pairedItem: { item: i }`
- **Watch trigger (0 inputs):** Output sets `pairedItem: { item: 0 }`
- All output items wrapped with `this.helpers.constructExecutionMetaData()`

From section **Error Handling**:

**DNS-Level Errors** — returned as structured data, not exceptions:
- NOERROR: normal output
- NXDOMAIN: normal output (critical for "domain appears" monitoring)
- SERVFAIL: normal output with warning
- REFUSED: normal output with warning
- FORMERR: throw `NodeOperationError` (indicates a bug)

**Transport-Level Errors:**
- Timeout: retry up to limit, then return item with `responseCode: "TIMEOUT"`
- Network unreachable: throw `NodeOperationError`
- Truncated response (TC flag): return partial data with `truncated: true`

**Continue on Fail:**
> The node respects `this.continueOnFail()`. When enabled, transport errors produce an output item with the error message rather than halting the workflow. DNS-level errors always produce structured output regardless.

## Tasks

- Set `pairedItem` on every output item in both Lookup and Watch nodes per the rules above
- Wrap all output items with `this.helpers.constructExecutionMetaData()`
- Implement DNS-level error handling: return response codes as structured data
- NXDOMAIN is not an error — return as normal structured output
- FORMERR throws `NodeOperationError`
- Implement transport-level error handling per the table above
- Implement `continueOnFail()` support

## Acceptance Criteria

- `pairedItem` is set on every output item in both nodes
- `constructExecutionMetaData()` wraps all output items
- NXDOMAIN produces structured output, not an exception
- SERVFAIL and REFUSED produce output with warnings
- `continueOnFail()` produces error items instead of halting
- Timeout produces `responseCode: "TIMEOUT"` after retries exhausted
