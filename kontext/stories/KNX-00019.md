Ensure full n8n linter compliance in strict mode and configure `package.json` for npm publishing and community node verification.

## Context

From the design document, section **n8n Linter Compliance**:

> The node will pass `eslint-plugin-n8n-nodes-base` in strict mode. Key rules:
> - `node-param-operation-without-no-data-expression` — Record Type and Resolver Mode set `noDataExpression: true`
> - `node-param-description-missing-from-dynamic-options` — All options include descriptions
> - `node-param-default-missing` — All parameters have explicit defaults
> - `node-param-display-name-miscased` — Display names use Title Case
> - `cred-class-field-name-missing-api` — Credential `name` ends with `Api`
> - `community-package-json-name-still-default` — Package name is `n8n-nodes-dns`

From section **Publishing and Verification**:

> The package will be published to npm under the name `n8n-nodes-dns` with the `n8n-community-node-package` keyword.

The `package.json` `n8n` object registers node and credential paths:
```json
{
  "n8n": {
    "n8nNodesApiVersion": 1,
    "nodes": [
      "dist/nodes/DnsLookup/DnsLookup.node.js",
      "dist/nodes/DnsWatch/DnsWatch.node.js"
    ],
    "credentials": [
      "dist/credentials/DnsServerApi.credentials.js"
    ]
  }
}
```

Keywords: `n8n-community-node-package`, `n8n`, `dns`, `domain`, `nameserver`, `propagation`, `dnsbl`, `spf`, `dmarc`, `dkim`, `email-security`, `monitoring`, `network`.

> The zero-dependency design, MIT license, `@n8n/node-cli` scaffolding, strict-mode linter compliance, and provenance attestation align with all current n8n community node verification criteria.

## Tasks

- Configure `package.json` with correct name (`n8n-nodes-dns`), keywords, `n8n` object, and MIT license
- Register node and credential paths under `dist/` in the `n8n` object
- Run `just lint` and fix all linter violations
- Verify all linter rules listed in the design document pass
- Run `just verify` (local `@n8n/scan-community-package` replica) and fix any issues
- Ensure zero runtime dependencies
- Create README.md with package description, installation instructions, usage examples, and configuration reference (linked from the codex `primaryDocumentation` field)

## Acceptance Criteria

- `just lint` passes with zero errors in strict mode
- `just verify` passes
- `package.json` contains all required keywords and correct `n8n` object
- Package has zero runtime dependencies
- README.md exists and is linked from the codex `primaryDocumentation` field
- All verification criteria from the design document are met
