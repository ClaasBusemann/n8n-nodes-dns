Implement the well-known resolver registry and system resolver parsing in `src/transport/dns-resolvers.ts`.

## Context

From the design document, section **Transport Layer > Well-Known Resolvers**:

> A static registry of public DNS resolvers. Stored as a constant — no external data fetching.

The registry includes: Cloudflare (1.1.1.1/1.0.0.1), Google (8.8.8.8/8.8.4.4), Quad9 (9.9.9.9/149.112.112.112), Quad9 Unfiltered, OpenDNS, Cloudflare Malware, Cloudflare Family, AdGuard, and Control D — with primary and secondary IPs for each.

> The "System" resolver mode reads nameservers from `/etc/resolv.conf` by parsing lines starting with `nameserver`.

From section **Architectural Constraint — File I/O**:

> This node has one controlled exception: reading `/etc/resolv.conf` in the System resolver mode. This is a read-only operation on a system configuration file [...] The functionality is isolated in `dns-resolvers.ts` and only invoked when the user explicitly selects System mode.

## Tasks

- Export a constant registry of well-known public DNS resolvers with IPs, names, and notes per the table in the design document
- Implement `parseResolvConf()` that reads `/etc/resolv.conf` and extracts nameserver IPs
- Ensure the `/etc/resolv.conf` read is isolated and only invoked for System mode
- Export a lookup function to resolve resolver mode + selection into target server IPs

## Acceptance Criteria

- Registry is a static constant with no external data fetching
- All 9 resolver entries from the design document are present with correct IPs
- System resolver parsing handles standard `resolv.conf` format
- File I/O is isolated to this single module
